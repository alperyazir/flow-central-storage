# Story 15.1: Nginx Gzip Compression + PGBouncer

## Status
Approved

## Story
**As a** system administrator,
**I want** response compression and database connection pooling configured,
**so that** API responses are smaller/faster and PostgreSQL connections are efficiently managed at scale.

## Acceptance Criteria
1. Nginx gzip enabled for JSON, text, CSS, JS, SVG responses
2. PGBouncer service running in docker-compose between API and PostgreSQL
3. API and worker connect to PGBouncer (port 6432) instead of PostgreSQL directly
4. PGBouncer health check passes
5. SQLAlchemy pool_size reduced (PGBouncer handles pooling now)
6. All existing functionality works through PGBouncer

## Dev Notes

### PGBouncer Reference (Flow Learn)
```ini
# /Users/alperyazir/Dev/dream-lms/pgbouncer/pgbouncer.ini
[databases]
* = host=db port=5432

[pgbouncer]
listen_addr = 0.0.0.0
listen_port = 6432
auth_type = scram-sha-256
auth_file = /etc/pgbouncer/userlist.txt
pool_mode = transaction
server_reset_query =
default_pool_size = 200
min_pool_size = 40
max_client_conn = 1500
```

For FCS, use smaller defaults: `default_pool_size=100`, `max_client_conn=500` (smaller system).

### Entrypoint Reference
```bash
#!/bin/sh
echo "\"${POSTGRES_USER}\" \"${POSTGRES_PASSWORD}\"" > /etc/pgbouncer/userlist.txt
exec pgbouncer /etc/pgbouncer/pgbouncer.ini
```

### Important
- PGBouncer `pool_mode=transaction` means connections return to pool after each transaction — compatible with SQLAlchemy
- `server_reset_query` should be empty when using transaction mode with no prepared statements
- Keep postgres healthcheck in docker-compose — pgbouncer depends on it
