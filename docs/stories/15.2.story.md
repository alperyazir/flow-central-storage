# Story 15.2: Fix N+1 Query + Optimize Material Stats

## Status
Approved

## Story
**As a** developer,
**I want** the teacher listing endpoint optimized to use bulk queries,
**so that** listing 100 teachers takes 3 queries instead of 401.

## Acceptance Criteria
1. `get_bulk_storage_stats()` method exists in material repository — single query for multiple teacher IDs
2. `get_storage_stats()` consolidated from 4 queries to 1-2 using conditional aggregation
3. `list_teachers()` endpoint uses bulk stats method
4. `list_trashed_teachers()` endpoint uses bulk stats method
5. API response schema unchanged (backward compatible)
6. All tests pass

## Dev Notes

### Current Code (N+1 pattern)
```python
# apps/api/app/routers/teachers_crud.py:116-133
for t in teachers:
    stats = _material_repository.get_storage_stats(db, t.id)  # 4 queries EACH
    items.append(TeacherListItem(..., material_count=stats["total_count"], ...))
```

### Target Code
```python
teacher_ids = [t.id for t in teachers]
bulk_stats = _material_repository.get_bulk_storage_stats(db, teacher_ids)
for t in teachers:
    stats = bulk_stats.get(t.id, {"total_count": 0, "total_size": 0, ...})
    items.append(TeacherListItem(..., material_count=stats["total_count"], ...))
```

### Conditional Aggregation Pattern
```python
select(
    func.count(Material.id).label("total_count"),
    func.coalesce(func.sum(Material.size), 0).label("total_size"),
    func.count(Material.id).filter(
        Material.file_type.in_(TEXT_MATERIAL_TYPES)
    ).label("ai_processable"),
    func.count(Material.id).filter(
        Material.ai_processing_status == "completed"
    ).label("ai_processed"),
).where(
    Material.teacher_id == teacher_id,
    Material.status != "archived"
)
```

### Key Files

### Important

- `AIProcessingStatusEnum.COMPLETED.value` for ai_processed filter
- Don't break the `by_type` breakdown — teachers detail page uses it
- Response schema `TeacherListResponse` must not change
