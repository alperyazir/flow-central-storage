# Story 15.3: Redis Cache Layer

## Status
Approved

## Story
**As a** developer,
**I want** frequently-read API data cached in Redis,
**so that** hot endpoints serve from cache instead of hitting the database on every request.

## Acceptance Criteria
1. Cache service exists with get/set/delete/invalidate_pattern methods
2. Books list endpoint cached (TTL: 5 min)
3. Book detail endpoint cached (TTL: 10 min)
4. Publishers list/detail endpoints cached (TTL: 10 min)
5. Teachers list endpoint cached (TTL: 5 min)
6. Teacher materials endpoint cached (TTL: 5 min)
7. Cache invalidated on POST/PUT/DELETE mutations for each resource
8. Cache key pattern: `fcs:{resource}:{id}:{params_hash}`
9. API response schemas unchanged

### Cache Service Pattern
```python
import json
import hashlib
from redis.asyncio import Redis

class CacheService:
    def __init__(self, redis: Redis):
        self._redis = redis

    async def get(self, key: str) -> dict | list | None:
        data = await self._redis.get(key)
        return json.loads(data) if data else None

    async def set(self, key: str, value: dict | list, ttl: int = 300):
        await self._redis.setex(key, ttl, json.dumps(value, default=str))

    async def delete(self, key: str):
        await self._redis.delete(key)

    async def invalidate_pattern(self, pattern: str):
        keys = []
        async for key in self._redis.scan_iter(match=pattern):
            keys.append(key)
        if keys:
            await self._redis.delete(*keys)
```

### Important
- Cache is READ-THROUGH: check cache → miss → query DB → store in cache → return
- Mutations ALWAYS invalidate first, then perform operation
- `invalidate_pattern` uses SCAN — safe for production (non-blocking)
- If Redis is down, cache.get returns None, cache.set is a no-op — app continues working
- Endpoints that accept pagination params need those in the cache key to avoid stale pages
